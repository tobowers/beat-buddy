1 What this app is (and isn't)

Beat Buddy is a browser game that answers one question: “Can you do this movement pattern with the beat for 20–60 seconds?” A child picks a rhythm game, watches an animated demo, then performs the pattern in front of the webcam while a metronome plays. The app detects approximate movement events (a clap, an opposite hand-to-knee touch, a starfish shape) and checks whether each one landed inside a forgiving timing window around the beat.

It supports an occupational-therapy-style home practice routine: rhythm, bilateral coordination, crossing the midline, motor planning, and sustained structured movement. The emotional-regulation benefit is indirect — building repeatable body skills during calm moments so the child has a stronger foundation over time.

Framing rules The app is a movement practice game, not a behavior-management tool. It never asks "are you calm?", never frames play as therapy or as fixing feelings, and is meant to be used when the child is already regulated enough to play — never during a meltdown or power struggle.

Technically, it does not certify perfect form. It verifies that the child is visible, is attempting the expected movement, is roughly in rhythm, alternates left/right roughly correctly, and can sustain the pattern for a short round. Success is deliberately achievable.

2 Architecture at a glance

Everything runs in the browser. The Bun server (index.ts) does one thing: serve index.html, which imports src/frontend.tsx. Bun's bundler transpiles the React/TypeScript on the fly (with hot reload in dev). There is no backend state, no database, no API.

Two independent clocks drive a round, and the scoring engine sits where they meet:

Key idea Movement events are timestamped with performance.now() as frames are processed, and every metronome beat is scheduled ahead of time with a known performance.now()-domain timestamp. Scoring is then just: did an event of the expected type land within ±400 ms of the beat's timestamp? No frame-rate coupling, no polling.

Mirror convention The demo figure and camera view are both mirrored like a bathroom mirror: the child's left hand corresponds to the limb on the viewer's left. 5 The metronome

src/metronome.ts uses the Web Audio API, not setInterval, for the actual tick timing. JavaScript timers jitter by tens of milliseconds under load; the audio hardware clock does not. The implementation is the classic lookahead scheduler:

  1. A coarse setInterval(25ms) loop wakes the scheduler.
  2. Each wake, it schedules every tick that falls within the next 150 ms on the AudioContext clock (osc.start(when)), where they play with sample-accurate timing regardless of main-thread jank.
  3. Accented ticks (988 Hz vs 660 Hz) mark beat 1 of each pattern repetition.

Bridging the two clocks

Pose events are timestamped with performance.now(), but audio runs on ctx.currentTime. The metronome computes the entire beat schedule up front, anchored in both domains at the same instant: 

const spb        = 60 / bpm;                     // seconds per beat
const startAudio = ctx.currentTime + 0.35;      // audio-clock anchor
const startPerf  = performance.now() + 350;     // same instant, perf-clock

// beat i plays at startAudio + i*spb  (audio domain)
// beat i "is"    at startPerf  + i*spb*1000 (perf domain — used by the Scorer)

Clock drift between the two domains over a 60-second round is far below the ±400 ms scoring window, so a single anchor point is sufficient. start() returns the full schedule as {index, perfTime}[]; visual beat callbacks fire via per-beat setTimeout (fine for UI, never used for scoring), and the Play screen ends the round with one timeout set to last beat + window + margin.

Count-in

Every round starts with 4 unscored count-in beats ("Get ready… 1 2 3 4") so the child hears the tempo before scoring begins. The scored beats are schedule.slice(4), each assigned its expected movement by cycling the pattern.

Sounds

  • Tick — a 90 ms sine blip with an exponential decay envelope; accented on pattern beat 1.
  • Hit blip — a soft two-note triangle arpeggio (E6→A6) played when a movement matches a beat, unless feedback mode is Quiet.

6 Pose estimation pipeline

src/pose.ts wraps MediaPipe Tasks Vision (@mediapipe/tasks-vision) with the pose_landmarker_lite model in runningMode: "VIDEO". The landmarker is a lazy singleton — the ~5 MB model and WASM runtime load once, on first use, from Google's CDN. It tries the GPU delegate first and falls back to CPU if WebGL isn't available.

The per-frame loop

const loop = () => {
  raf = requestAnimationFrame(loop);
  if (video.currentTime === lastTime) return;   // skip duplicate frames
  lastTime = video.currentTime;

  const now = performance.now();
  const result = landmarker.detectForVideo(video, now);
  const lm = result.landmarks?.[0];             // 33 normalized landmarks

  drawSkeleton(ctx, lm, w, h, color);            // canvas overlay
  for (const ev of detector.update(lm, now))   // movement events
    scorer.addEvent(ev, now);
};

Landmarks used

MediaPipe returns 33 body landmarks in normalized coordinates (x, y ∈ [0,1] relative to frame size, plus a visibility confidence). Beat Buddy uses: nose (0), shoulders (11/12), elbows (13/14), wrists (15/16), hips (23/24), knees (25/26), ankles (27/28).

Full-body visibility check

fullBodyVisible() requires nose, shoulders, hips, knees, and ankles to all have visibility > 0.5 and coordinates comfortably inside the frame. The Setup screen turns green when it passes; the Play screen shows a friendly "Step back so I can see you!" pill when it fails mid-round (scoring simply finds no events while the child is out of frame — there is no penalty logic).

Why pose only, no hands model Pose landmarks alone are enough for cross-crawls, starfish, and knee events. Claps are approximated as "wrists snap together in front of the torso" — less precise than the MediaPipe Hands model but far cheaper, and well within the app's tolerance-first design. Microphone-based clap detection was deliberately skipped to keep the privacy story simple.

7 Movement event detection

src/detect.ts converts a stream of landmark frames into discrete, timestamped events. Three mechanisms keep it robust:

  • Torso normalization. Every distance threshold is expressed as a fraction of torso length (average shoulder→hip distance), so detection works whether the child is 2 m or 4 m from the camera.
  • Edge triggering with hysteresis. Each event fires once when its condition enters (tight threshold) and re-arms only after it clearly releases (wider threshold). Holding a pose can't spam events.
  • Refractory period. A 300 ms minimum gap between firings of the same event type absorbs landmark jitter.

// The shared edge-trigger helper — one boolean pair per event:
const check = (ev, entered, released) => {
  if (armed[ev] && entered && now - lastFired[ev] > 300) {
    armed[ev] = false; lastFired[ev] = now; fired.push(ev);
  } else if (!armed[ev] && released) {
    armed[ev] = true;          // hysteresis: re-arm only on clear release
  }
};

Frames with fewer than 33 landmarks, or an implausibly small torso (< 0.05 in normalized units — usually a spurious detection), are ignored entirely.

8 Scoring engine

src/scorer.ts receives the scored-beat schedule (index, perf-domain timestamp, expected movement) at round start. Every incoming event is matched greedily:

  1. Consider only beats whose expected equals the event type.
  2. Skip beats that are already matched (each beat is satisfied at most once).
  3. Among the rest, pick the beat with the smallest |event − beat| offset, if it is within the timing window.

The window is the spec's Easy mode: ±400 ms (a single constant, TIMING_WINDOW_MS in src/types.ts — Normal ±300 and Hard ±200 are a one-line change, planned as a parent setting). At 50 BPM a beat is 1200 ms apart, so windows never overlap for the same movement type in practice.

finalize() — called by the Play screen once the last window has closed — returns hits, total, the longest consecutive-hit streak, and per-beat results including the signed timing offset of every hit (useful later for a parent-facing "tends to rush / tends to drag" view).

9 Feedback philosophy

Feedback is deliberately asymmetric: hits are celebrated, misses are silent.

  • During a round, a matched beat produces a star pop and a soft blip. A missed beat produces… nothing. There is no "wrong", "missed", or red flash anywhere.
  • Gentle mode rotates neutral encouragements ("Nice, keep going!", "Follow the beat!") on a timer — they are not tied to performance, so they can't become a judgment.
  • Result tiers are all positive: ≥90% "Superstar!", ≥70% "Awesome rhythm!", ≥50% "Great practice!", below that "Good try!" with "Every round makes your body smarter." The lowest tier still earns a star.
  • "Try it slower" is offered as an appealing next step, not a remediation.
  • Score numbers appear only when Score mode is explicitly chosen; parents can turn all numbers off (Gentle shows kid-phrased counts, Quiet shows none).
Success criteria For this use case success should be achievable: roughly 70% of beats on Easy is a "successful round". The forgiving window, torso-scaled thresholds, and count-in all serve that goal.

10 Privacy model

  • All camera processing happens locally in the browser tab — MediaPipe runs as WASM/WebGL on-device. No video frame ever leaves the machine.
  • No video or audio is recorded or stored. The microphone is never requested (audio: false in the getUserMedia constraints).
  • The camera stream is a module-level singleton, and it is fully stopped (track.stop()) whenever the child returns to the game picker — the browser's camera indicator light goes off.
  • The only persisted data is the settings object (speed / length / feedback) in localStorage. No session history yet; when added it will also be local-only.
  • No analytics, no third-party requests at runtime except the one-time model/WASM download from Google's CDN and Google Fonts.
  • Nothing identifies the child; there are no accounts and no names.

The parent-facing copy on the main screen reflects exactly this: "Camera video stays on this device. The app uses the camera only to spot body landmarks during the game. Nothing is saved or uploaded."

12 Testing

The pure logic — the parts that decide whether a child "did it" — is unit tested with bun test:

  • Scorer (src/scorer.test.ts): in-window matches, out-of-window rejections, wrong-type rejections, one-event-per-beat, nearest-beat selection, early events, streak computation, signed offsets.
  • EventDetector (src/detect.test.ts): synthetic 33-landmark frames verify each rule fires, doesn't re-fire while held, re-arms after release, and that hands-overhead is not a clap.

The full loop (camera → round → result) was verified end-to-end in a headless Chromium with --use-fake-device-for-media-stream: the metronome schedule, count-in, round completion, and result screen all work with zero console errors. The fake feed contains no person, which also exercises the "no landmarks" path safely.

13 Tuning & extending

If detection feels wrong in real use

SymptomKnob (all in src/detect.ts unless noted)
Knee touches not registeringRaise the enter threshold 0.45 toward 0.55
Touches double-countingWiden the release threshold 0.7, or raise REFRACTORY_MS
Claps missedRaise 0.32, or loosen the chest-height band
Starfish too hard to hitLower arm-out 0.5 or feet-wide ratio 1.35
Everything feels too strict/loose overallTIMING_WINDOW_MS in src/types.ts (400 → 300/200 for Normal/Hard)

Adding a game

  1. Add a GamePattern entry in src/games.ts (id, color, steps).
  2. If it needs a new movement: add the variant to MovementEvent in src/types.ts, a detection rule in EventDetector.update(), a figure pose in src/Figure.tsx's POSES, and a preview mapping in Pick.tsx.
  3. That's it — demo, play, and scoring are entirely pattern-driven.

Planned next phases (from the spec)

  • Copy Me — the app plays a 4-beat pattern, the child repeats it; needs a listen-then-perform round structure but reuses the detector and scorer as-is.
  • Stomp-Clap — stomps are weak with pose alone; the plan is a "march" variant (knee-lift detection) first, optionally microphone impact detection later.
  • Parent customization — timing tolerance picker, per-game BPM, hidden score, local session history with manual export. The scorer already records per-beat offsets to support this.
